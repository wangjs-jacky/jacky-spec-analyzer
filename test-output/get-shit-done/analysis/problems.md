# GSD 问题分析

## 1. 核心问题

> **Context Rot（上下文腐化）**：随着对话进行，Claude 的上下文窗口被填充，导致输出质量下降。

**问题描述**: 当你与 Claude 长时间对话时，它会"变笨"——开始遗忘早期信息、重复自己、输出质量下降。这不是模型问题，而是上下文管理问题。

**影响范围**: 所有依赖 AI 编程的开发者

## 2. 背景与现状

### 2.1 之前的做法

1. **Vibecoding（随意编码）**: 描述想要什么 → AI 生成代码 → 得到不一致的垃圾
2. **手动上下文管理**: 频繁清除对话、复制粘贴关键信息
3. **传统 Spec 工具**: BMAD、Speckit 等，但过于复杂（冲刺会议、故事点、Jira 工作流）

### 2.2 现有方案的痛点

| 痛点 | 描述 | 根本原因 | 影响 |
|------|------|----------|------|
| 上下文腐化 | 对话越长，质量越差 | Claude 的上下文窗口有限 | 产出不稳定 |
| Vibecoding 不可靠 | 描述需求 → 垃圾代码 | 缺乏规范和上下文工程 | 大规模化时崩溃 |
| 传统工具太复杂 | 企业流程、角色扮演 | 为大团队设计 | 独立开发者不适用 |
| 缺乏全局理解 | AI 只看到片段 | 没有项目级上下文文件 | 实现与愿景不符 |
| 无原子提交 | 大批量代码提交 | 没有任务级跟踪 | 难以回滚和调试 |

## 3. 问题根源分析

### 3.1 上下文窗口限制

Claude 的上下文窗口有限（约 200K tokens）。随着对话进行：
- 早期信息被"遗忘"
- 上下文被无关信息污染
- 模型开始"节省"输出

### 3.2 缺乏上下文工程

大多数人只给 Claude 代码，不给：
- 项目愿景
- 技术决策背景
- 需求追踪
- 历史决定

### 3.3 工作流设计问题

- 没有规范的阶段划分
- 没有验证机制
- 没有状态跟踪
- 提交粒度过大

## 4. GSD 的解决思路

```
问题: 上下文腐化
解决: 每个任务使用全新的 200K 上下文窗口

问题: 缺乏上下文
解决: PROJECT.md + REQUIREMENTS.md + ROADMAP.md + STATE.md

问题: 工作流混乱
解决: 讨论 → 规划 → 执行 → 验证 循环

问题: 提交粒度大
解决: 每个任务原子提交
```
