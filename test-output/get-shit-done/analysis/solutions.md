# GSD 解决方案

## 1. 核心思路

> **上下文工程 + 多代理编排 = 可靠的 AI 编程**

通过：
1. 为每个任务提供全新的 200K 上下文
2. 规范化的项目文档体系
3. 结构化的工作流程
4. 原子级别的 git 提交

## 2. 工作流程

### 流程图

```mermaid
graph TB
    A[/gsd:new-project] --> B[提问 → 研究 → 需求 → 路线图]
    B --> C[/gsd:discuss-phase N]
    C --> D[捕获实现偏好]
    D --> E[/gsd:plan-phase N]
    E --> F[研究 → 规划 → 验证]
    F --> G[/gsd:execute-phase N]
    G --> H[Wave 执行]
    H --> I[/gsd:verify-work N]
    I --> J{通过?}
    J -->|否| K[诊断 + 修复]
    K --> G
    J -->|是| L{还有阶段?}
    L -->|是| C
    L -->|否| M[/gsd:complete-milestone]
```

## 3. 关键设计决策

| 决策 | 原因 | 权衡 |
|------|------|------|
| 每个任务使用新上下文 | 避免上下文腐化 | 启动开销增加 |
| XML 格式的任务计划 | 对 Claude 最优解析 | 人类可读性降低 |
| 原子提交 | 可追溯、可回滚 | 提交数量增加 |
| 并行 Wave 执行 | 加速开发 | 依赖管理复杂 |
| 项目文档体系 | 全局上下文 | 维护成本 |

## 4. 与现有方案对比

| 维度 | Vibecoding | 传统 Spec 工具 | GSD |
|------|------------|----------------|-----|
| 上下文管理 | 无 | 有 | ✅ 200K 新鲜上下文 |
| 学习曲线 | 低 | 高 | 中 |
| 复杂度 | 低 | 高（企业流程） | 低 |
| 原子提交 | 无 | 有 | ✅ 每任务提交 |
| 多代理编排 | 无 | 无 | ✅ Wave 执行 |
| 适合团队 | 无 | 大团队 | 独立开发者/小团队 |

## 5. 文件体系

```
.planning/
├── PROJECT.md        # 项目愿景（始终加载）
├── REQUIREMENTS.md   # 范围化的 v1/v2 需求
├── ROADMAP.md        # 阶段分解和状态
├── STATE.md          # 决定、阻碍、会话记忆
├── config.json       # 工作流配置
├── research/         # 领域研究
├── todos/            # 待办事项
└── phases/
    └── XX-phase-name/
        ├── CONTEXT.md      # 实现偏好
        ├── RESEARCH.md     # 生态研究
        ├── XX-YY-PLAN.md   # 原子执行计划
        ├── XX-YY-SUMMARY.md # 执行结果
        └── VERIFICATION.md  # 验证报告
```
