# optimize-skill

系统化优化 Claude Code Skill 的质量，通过迭代测试和反馈改进 skill 的提示词。

## 使用方式

```bash
# 优化指定的 skill
/optimize-skill <skill-name>

# 优化时使用特定测试案例
/optimize-skill spec-analyzer --test-case https://github.com/gsd-build/get-shit-done

# 指定优化目标
/optimize-skill spec-analyzer --focus output-quality
```

## 参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--test-case, -t` | 测试案例（仓库 URL 或本地路径） | 无 |
| `--focus, -f` | 优化焦点（output-quality/prompt-clarity/error-handling） | `output-quality` |
| `--iterations, -i` | 最大迭代次数 | `3` |

## 工作流程

### 1. 读取目标 Skill

首先读取要优化的 skill 文件：

```bash
# Skill 通常位于以下位置之一：
# - ./skills/<skill-name>/SKILL.md
# - ~/.claude/skills/<skill-name>/SKILL.md
# - $JACKY_SKILLS_DIR/<skill-name>/SKILL.md
```

### 2. 分析 Skill 结构

检查 skill 的以下方面：

- **元数据**: `name` 和 `description` 是否准确？
- **触发条件**: 描述是否能让 Claude 正确判断何时使用？
- **工作流程**: 步骤是否清晰、可执行？
- **输出格式**: 是否定义了明确的输出结构？
- **示例**: 是否提供了足够的使用示例？

### 3. 执行测试案例（如果提供）

如果提供了 `--test-case`，执行以下步骤：

1. 在新会话中调用目标 skill
2. 记录输出结果
3. 分析输出质量

### 4. 识别问题

根据测试结果，识别以下类型的问题：

#### 输出质量问题

- [ ] 输出是否遗漏关键信息？
- [ ] 输出格式是否一致？
- [ ] 输出深度是否足够？
- [ ] 输出是否包含不必要的冗余？

#### 提示词清晰度问题

- [ ] 步骤描述是否清晰？
- [ ] 是否有歧义或模糊的指令？
- [ ] 条件判断是否明确？
- [ ] 边界情况是否覆盖？

#### 错误处理问题

- [ ] 输入验证是否充分？
- [ ] 异常情况是否有处理方案？
- [ ] 错误信息是否友好？

### 5. 生成优化建议

为每个识别的问题生成具体的优化建议：

```markdown
## 优化建议

### 问题 1: 输出遗漏核心概念

**位置**: "提取技术要点" 章节

**当前内容**:
> 提取关键技术实现细节

**建议修改**:
> 提取关键技术实现细节，包括：
> - 核心算法和数据结构
> - 关键 API 和接口设计
> - 重要的配置和参数
> - 与其他系统的集成点

**原因**: 原描述太笼统，容易导致遗漏重要信息
```

### 6. 应用优化

在用户确认后，应用优化修改：

1. 更新 SKILL.md 文件
2. 保留修改历史注释
3. 更新版本号（如果有）

### 7. 验证改进

重新运行测试案例，对比优化前后的输出：

```markdown
## 验证结果

| 维度 | 优化前 | 优化后 | 改进 |
|------|--------|--------|------|
| 信息完整性 | 60% | 90% | +30% |
| 格式一致性 | 70% | 95% | +25% |
| 深度 | 中等 | 深 | ↑ |
```

## 优化模式

### output-quality（输出质量）

专注于提升 skill 的输出质量：

1. 检查输出模板是否完整
2. 检查提取规则是否精确
3. 检查格式要求是否严格
4. 添加输出质量检查清单

### prompt-clarity（提示词清晰度）

专注于提升提示词的清晰度：

1. 消除歧义表述
2. 添加具体示例
3. 细化步骤说明
4. 增加条件判断

### error-handling（错误处理）

专注于提升错误处理能力：

1. 添加输入验证
2. 添加异常处理流程
3. 添加友好的错误提示
4. 添加降级方案

## Skill 质量检查清单

### 元数据检查

- [ ] `name` 简洁且唯一
- [ ] `description` 准确描述触发条件
- [ ] description 包含足够的触发关键词

### 结构检查

- [ ] 有清晰的章节划分
- [ ] 有明确的使用方式说明
- [ ] 有参数说明（如有参数）
- [ ] 有工作流程说明
- [ ] 有示例（推荐）

### 内容检查

- [ ] 步骤描述可执行
- [ ] 条件判断明确
- [ ] 输出格式定义清晰
- [ ] 边界情况有说明
- [ ] 有常见问题解答（可选）

### 可维护性检查

- [ ] 有版本历史或修改记录
- [ ] 结构便于后续扩展
- [ ] 避免重复内容

## 示例

### 优化 spec-analyzer

```bash
/optimize-skill spec-analyzer --test-case https://github.com/gsd-build/get-shit-done
```

**发现的问题**:
1. "提取技术要点" 描述太笼统
2. 缺少对 XML 格式文档的处理说明
3. 输出格式缺少数据流描述

**应用的优化**:
1. 细化技术要点提取规则
2. 添加 XML 文档解析指令
3. 在模板中添加数据流章节

### 优化自定义 skill

```bash
/optimize-skill my-custom-skill --focus prompt-clarity
```

**发现的问题**:
1. "分析代码" 步骤有歧义
2. 缺少对大型项目的处理说明

**应用的优化**:
1. 明确 "分析代码" 的具体动作
2. 添加分批处理的说明

## 注意事项

1. **迭代优化**: 一次只解决 1-2 个核心问题，避免过度修改
2. **保留历史**: 使用注释保留重要的修改历史
3. **测试验证**: 每次优化后都要验证效果
4. **用户确认**: 重大修改前需要用户确认
5. **版本控制**: 建议使用 git 管理 skill 文件
