# GSD 解决方案

## 1. 核心思路

> **把复杂性放在系统中，而不是用户工作流中**

GSD 通过精心设计的上下文工程、XML Prompt 格式和多 Agent 编排，让用户只需要：
1. 描述想要什么
2. 确认计划
3. 验证结果

系统在背后处理所有复杂性：上下文管理、任务分解、并行执行、自动验证。

## 2. 工作流程

### 流程图

```mermaid
flowchart TB
    subgraph Init["初始化阶段"]
        A[/gsd:new-project] --> B[提问理解需求]
        B --> C[研究技术栈]
        C --> D[生成需求文档]
        D --> E[创建路线图]
    end

    subgraph Phase["迭代阶段"]
        F[/gsd:discuss-phase] --> G[捕获实现偏好]
        G --> H[/gsd:plan-phase]
        H --> I[研究实现方案]
        I --> J[生成原子计划]
        J --> K[验证计划完整性]
        K --> L[/gsd:execute-phase]
        L --> M[Wave 并行执行]
        M --> N[每个任务独立上下文]
        N --> O[原子提交]
        O --> P[/gsd:verify-work]
        P --> Q{用户验收}
        Q -->|通过| R[下一阶段]
        Q -->|失败| S[生成修复计划]
        S --> L
    end

    E --> F
    R --> F
```

### 步骤说明

1. **初始化** (`/gsd:new-project`)
   - 系统提问直到完全理解你的想法
   - 并行研究技术栈、特性、架构、陷阱
   - 提取 v1/v2/超出范围的需求
   - 创建映射到需求的路线图

2. **讨论阶段** (`/gsd:discuss-phase`)
   - 路线图只有一两句话描述
   - 这个步骤捕获你的实现偏好
   - 生成 CONTEXT.md 指导后续研究

3. **计划阶段** (`/gsd:plan-phase`)
   - 研究如何实现这个阶段
   - 创建 2-3 个原子任务计划
   - 验证计划是否覆盖需求

4. **执行阶段** (`/gsd:execute-phase`)
   - Wave 并行执行独立任务
   - 每个计划用独立的 200K 上下文
   - 每个任务完成立即提交

5. **验证阶段** (`/gsd:verify-work`)
   - 提取可测试的交付物
   - 逐个引导用户测试
   - 失败时自动诊断并生成修复计划

## 3. 关键设计决策

| 决策 | 原因 | 权衡 |
|------|------|------|
| **200K 新鲜上下文** | 避免上下文腐化 | 需要更多 API 调用 |
| **XML Prompt 格式** | 结构化、精确、无歧义 | 学习成本（但系统处理） |
| **Wave 并行执行** | 提高效率，减少等待 | 需要正确识别依赖关系 |
| **原子提交** | 可追溯、可回滚 | 提交数量多 |
| **需求 ID 追踪** | 确保需求被实现 | 需要维护映射关系 |
| **多 Agent 编排** | 隔离上下文，提高质量 | 系统复杂度增加 |

## 4. 与现有方案对比

| 维度 | Vibecoding | 企业 Spec 工具 | GSD |
|------|------------|----------------|-----|
| **上下文管理** | 无 | 部分支持 | ✅ 200K 新鲜上下文 |
| **流程复杂度** | 低 | 高（Sprint/Jira） | 低（几个命令） |
| **需求追踪** | 无 | 有 | ✅ 需求 ID + Traceability |
| **并行执行** | 无 | 通常无 | ✅ Wave 执行 |
| **验证机制** | 无 | 人工 | ✅ 自动 + UAT |
| **适合团队规模** | 1人 | 10+ 人 | 1-3 人 |

## 5. Wave 执行机制

### 工作原理

```
┌─────────────────────────────────────────────────────────────────────┐
│  PHASE EXECUTION                                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  WAVE 1 (parallel)          WAVE 2 (parallel)          WAVE 3       │
│  ┌─────────┐ ┌─────────┐    ┌─────────┐ ┌─────────┐    ┌─────────┐ │
│  │ Plan 01 │ │ Plan 02 │ →  │ Plan 03 │ │ Plan 04 │ →  │ Plan 05 │ │
│  │         │ │         │    │         │ │         │    │         │ │
│  │ User    │ │ Product │    │ Orders  │ │ Cart    │    │ Checkout│ │
│  │ Model   │ │ Model   │    │ API     │ │ API     │    │ UI      │ │
│  └─────────┘ └─────────┘    └─────────┘ └─────────┘    └─────────┘ │
│       │           │              ↑           ↑              ↑       │
│       └───────────┴──────────────┴───────────┘              │       │
│              Dependencies: Plan 03 needs Plan 01            │       │
│                          Plan 04 needs Plan 02              │       │
│                          Plan 05 needs Plans 03 + 04        │       │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 规则

- **独立计划** → 同一波次 → 并行执行
- **依赖计划** → 后续波次 → 等待依赖完成
- **文件冲突** → 顺序执行或合并到同一计划

### 最佳实践

**推荐：垂直切片**
```
Plan 01: User (model + API + UI)
Plan 02: Product (model + API + UI)
```

**避免：水平分层**
```
Plan 01: All models
Plan 02: All APIs
Plan 03: All UIs
```
