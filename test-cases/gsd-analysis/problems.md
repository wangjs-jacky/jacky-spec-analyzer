# GSD 问题分析

## 1. 核心问题

> **Context Rot（上下文腐化）**：随着 Claude 填充上下文窗口，输出质量逐渐下降。

**问题描述**：在长对话中，Claude 的上下文窗口会被历史对话填满，导致：
- 输出变得冗长但内容空洞（"我会更简洁"）
- 遗忘早期做出的决策
- 偏离原始需求
- 产生不一致的代码风格
- 出现幻觉或错误信息

**影响范围**：所有使用 Claude Code 进行大型项目开发的用户

## 2. 背景与现状

### 2.1 之前的做法

在没有 GSD 之前，用户主要有两种方式：

1. **Vibecoding（直觉编码）**
   - 描述想要什么，AI 生成代码
   - 没有结构化流程
   - 结果不一致，难以扩展

2. **其他 Spec-driven 工具**
   - BMAD、Speckit、Taskmaster 等
   - 但它们过于复杂（冲刺仪式、故事点、Jira 工作流）
   - 不适合独立开发者

### 2.2 现有方案的痛点

| 痛点 | 描述 | 根本原因 | 影响 |
|------|------|----------|------|
| **Vibecoding 不可靠** | 没有规范驱动，输出质量不稳定 | 缺乏结构化上下文管理 | 大型项目无法维护 |
| **企业工具过度工程** | Sprint、故事点、Stakeholder 会议 | 模仿大公司流程 | 独立开发者不需要这些 |
| **上下文腐化** | 长对话质量下降 | 上下文窗口有限且会被污染 | 最终输出偏离预期 |
| **缺乏追踪** | 需求和实现脱节 | 没有需求 ID 追踪机制 | 难以验证是否完成 |
| **无验证机制** | 代码写完就算完成 | 缺少自动化验证 | Bug 和遗漏被发现太晚 |

## 3. 问题根源分析

### 3.1 上下文窗口限制

```
┌─────────────────────────────────────────────────────────┐
│              上下文窗口生命周期                          │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  开始 ──────────────────────────────────────────→ 结束  │
│   │                                                │     │
│   │  新鲜、精准              冗余、混乱、质量下降  │     │
│   │     ↓                          ↓              │     │
│   │  [高质量输出]            [低质量输出]         │     │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**根本原因**：LLM 的上下文窗口是有限的，当被历史对话填满后：
1. 新输入被压缩
2. 模型"记住"太多无关信息
3. 注意力机制被分散

### 3.2 缺乏结构化流程

**Vibecoding 的问题**：
- 没有需求文档 → 范围蔓延
- 没有追踪机制 → 不知道完成了什么
- 没有验证步骤 → 问题积累

**企业工具的问题**：
- 复杂性在用户工作流中，而不是在系统中
- 需要学习大量概念
- 不适合 1-2 人的小团队

### 3.3 Agent 编排缺失

**没有专门的编排系统**：
- 单一 Agent 处理所有任务
- 没有并行执行能力
- 无法隔离上下文

## 4. 问题总结

GSD 要解决的核心问题是：

1. **如何保持上下文新鲜？** → 每个 Plan 用独立的 200K 上下文
2. **如何确保需求被实现？** → 需求 ID 追踪 + must_haves 验证
3. **如何提高执行效率？** → Wave 并行执行 + 专门 Agent
4. **如何验证工作完成？** → 自动化验证 + 用户验收测试
