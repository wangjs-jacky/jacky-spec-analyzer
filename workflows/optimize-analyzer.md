# spec-analyzer 优化工作流

## 目的

通过迭代优化 spec-analyzer 的输出质量，使其能够准确提取 spec-driven 仓库的核心知识。

## 工作流程

### 1. 运行分析

```bash
/spec-analyzer <target-repo> --depth deep
```

### 2. 审查输出

检查以下方面：

- [ ] 问题域是否准确提取？
- [ ] 痛点分析是否全面？
- [ ] 解决方案描述是否清晰？
- [ ] 技术要点是否遗漏关键内容？
- [ ] 核心概念定义是否准确？
- [ ] 工作流程是否完整？

### 3. 标注问题

在输出文件中添加注释：

```markdown
<!-- ISSUE: 这里应该提到 Wave 并行执行的概念 -->
<!-- ISSUE: 遗漏了需求追踪机制的说明 -->
<!-- ISSUE: 核心概念定义不够清晰 -->
```

### 4. 更新模板

根据问题更新 `templates/analysis-template.md`：

```markdown
# 更新模板示例

## 问题域

### 核心问题
<!-- 添加: 必须提取项目要解决的核心问题 -->

### 现有痛点
<!-- 添加: 列出之前方案的至少 3 个痛点 -->
```

### 5. 更新 Skill

更新 `skills/spec-analyzer/SKILL.md` 中的分析指令：

1. **添加新的分析维度**: 如果发现遗漏了重要的分析维度
2. **细化提取规则**: 如果提取的内容不够精确
3. **优化输出格式**: 如果输出格式不够清晰

### 6. 验证改进

重新运行分析，验证输出质量是否提升：

```bash
/spec-analyzer <target-repo> --depth deep --output ./analyzed-v2
```

对比两个版本的输出，确认改进有效。

## 质量标准

### 问题域质量标准

- [ ] 核心问题描述清晰，一句话能概括
- [ ] 至少列出 3 个现有方案的痛点
- [ ] 痛点分析包含：描述、根本原因、影响
- [ ] 问题根源分析深入，不是表面描述

### 解决方案质量标准

- [ ] 核心思路一句话能概括
- [ ] 工作流程有 Mermaid 图
- [ ] 至少列出 3 个关键设计决策
- [ ] 每个决策说明：原因、权衡
- [ ] 与现有方案有对比表格

### 技术要点质量标准

- [ ] 核心概念有定义和用途
- [ ] 关键实现有代码示例
- [ ] 数据流描述清晰
- [ ] 扩展点说明完整

## 常见问题

### Q: 分析结果太浅，没有深度？

**原因**: 可能是分析模式选择 quick 而不是 deep

**解决**: 使用 `--depth deep` 参数，或更新 SKILL.md 中的深度分析指令

### Q: 遗漏了关键概念？

**原因**: 扫描文档时可能遗漏了某些文件

**解决**: 检查是否扫描了所有 `.md` 文件，更新扫描规则

### Q: 输出格式不一致？

**原因**: 模板定义不够严格

**解决**: 更新 `templates/analysis-template.md`，添加更严格的格式要求

## 示例：优化 GSD 分析

### 第一轮分析

```bash
/spec-analyzer https://github.com/gsd-build/get-shit-done --depth deep
```

### 发现问题

1. 遗漏了 "Context Rot" 概念的定义
2. Wave 执行机制描述不够详细
3. 没有提取 XML Prompt 格式的细节

### 更新 Skill

在 `SKILL.md` 的"分析重点"部分添加：

```markdown
### 4. 技术细节提取

- **XML Prompt 格式**: 提取 `<task>` 标签的结构和各字段含义
- **Wave 执行**: 详细描述并行和串行的规则
- **上下文管理**: 解释 200K 新鲜上下文的含义
```

### 第二轮分析

重新运行，验证改进效果。
