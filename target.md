# GSD 工作流核心知识点

## 一、AI 编程工作流的重要性

在没有规范工作流的情况下进行AI编程，常见的问题包括：
- **上下文过长**：随着对话增长，AI模型会产生幻觉、走捷径，遗忘之前的约束条件
- **需求分散**：功能要求、UI调整、验证规则等散落在各处，无法形成统一约束
- **主代理负载过重**：所有对话在主agent进行，占用大量上下文
- **会话丢失**：关闭对话或新建会话后，所有上下文需要重新开始

## 二、GSD 工作流概述

**GSD**（GitHub Stars约3万）是一个**规范驱动的开源AI编程工作流**，通过30+个命令将项目开发流程切分为精细化阶段。

**核心解决机制：**

| 问题 | GSD解决方案 |
|------|------------|
| 超长对话 | 开启子代理，将需求拆分为多个可并行执行的计划 |
| 文件管理 | 生成结构化文件体系，文件间存在依赖关系，可同步更新状态 |
| 执行效率 | 标记可并行/需串行的任务，提升编码效率，减少上下文占用 |

## 三、GSD 文件体系

GSD通过大量文件伴随整个开发过程，分为**全局文件**和**阶段文件**。

**初始化阶段生成的核心全局文件（`new project`命令）：**

| 文件名 | 作用 |
|--------|------|
| `project.md` | 项目描述、主要功能、MVP阶段目标 |
| `checklist.md` | 验收清单，根据列出的功能制定计划 |
| `roadmap.md` | 路线图，将需求分阶段，与功能关联 |
| `state.md` | 项目阶段运行情况，记录完成百分比，支持断点续传 |

## 四、GSD 完整工作流程

```
初始化项目 → 需求讨论 → 生成计划 → 执行编码 → 验收测试 → [创建里程碑/新增阶段循环]
```

### 1. 初始化项目（`new project`）
- **已有代码项目**：先执行`map codebase`，扫描代码库生成技术栈、架构、目录结构说明
- **全新项目**：生成上述4个全局核心文件

### 2. 需求讨论（`discuss [阶段编号]`）
- **可跳过条件**：若`new project`时已提供完整PRD文档，可直接跳过
- **执行过程**：通过工具向用户提问，收集信息后生成`context.md`（需求确认文档）
- **输出**：阶段分析结果，明确每个阶段要做什么

### 3. 生成计划（`plan [阶段编号]`）
- 调用子agent进行分析、设计
- 生成具体可执行的`plan-[编号].md`文件
- **关键设计**：计划文档中包含`way`编号，用于后续并行/串行判断

### 4. 执行编码（`act [阶段编号]`）
- **核心机制**：分析所有plan文档的`way`编号
  - **相同`way`编号**：放入同一子代理**串行**执行
  - **不同`way`编号**：开启**多个子代理并行**执行
- **依赖处理**：plan文档中明确依赖关系（如"依赖02-1完成后执行"）

### 5. 验收测试（`verify [阶段编号]`）
GSD提供两种测试方式：

| 方式 | 命令 | 特点 |
|------|------|------|
| **人为参与（推荐）** | `verify` | 列出测试点，用户手动验证后反馈结果（pass/问题），AI修复后继续 |
| **自动测试** | `add test` | 根据plan生成测试用例并自动执行，适合测试特定方法/流程 |

## 五、进阶功能

### 1. 里程碑（`milestone`）
- 用于标记**较大的迭代节点**
- 内部执行逻辑类似`new project`

### 2. 新增阶段（`gsd`相关命令）
- 在现有需求中增加新阶段
- 重启完整的"讨论→计划→执行→验收"流程

### 3. 快速模式（`quick`）
- **适用场景**：简短开发、简单功能修改
- **特点**：跳过discuss和验收阶段，直接修改代码
- **输出**：生成`quick.md`文件，便于后续追溯

## 六、GSD 的优缺点分析

### ✅ 优势
- 完整的流程追踪和文档体系
- 支持并行开发，提升效率
- 断点续传，状态可恢复
- 适合中大型项目的持续迭代

### ❌ 局限与注意事项

| 局限 | 说明 |
|------|------|
| **Token消耗高** | 生成大量文件，开启多个子代理 |
| **用户参与度低** | 流程自动化程度高，中途难以插入干预 |
| **学习成本高** | 30+个命令，需研究源码理解用法 |
| **依赖信息质量** | 需用户清晰描述需求，否则"垃圾进垃圾出" |
| **执行时间长** | 受限于模型生成代码/文档的速度 |

### 🎯 适用场景
- **中大型项目**或功能
- 需要**持续迭代**的开发
- 团队已有**规范的开发流程习惯**

### ⚠️ 使用建议
- 日常简单开发使用`quick`模式
- 中大型功能使用完整流程
- 务必区分使用场景，避免流程误用导致时间浪费
